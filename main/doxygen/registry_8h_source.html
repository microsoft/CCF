24 // By subclassing DynamicJSEndpointRegistry, an application gains the
25 // ability to execute custom JavaScript endpoints, and exposes the ability to
26 // install them via install_custom_endpoints(). The JavaScript code for these
27 // endpoints is stored in the internal KV store under a namespace configured
28 // in the second argument to the constructor. Calling
29 // install_custom_endpoints() is logically equivalent to passing a set_js_app
30 // proposal in governance, and the payload format is currently identical,
31 // except the controlling logic resides in the application space.
32 //
33 // Known limitations:
34 //
35 // No auditability yet, COSE Sign1 auth is recommended, but the signature is
36 // not stored.
37 // No support for historical endpoints yet.
38 // No support for import from external modules.
39 //
40 // Additional functionality compared to set_js_app:
41 //
42 // The KV namespace can be private, to keep the application confidential if
43 // desired.
